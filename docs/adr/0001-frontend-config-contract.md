# ADR-0001: Frontend config is one platform-owned contract, resolved by field knowability

- **Status:** Proposed (2026-07-12)
- **Deciders:** Hans Kristian Flaatten (platform)
- **Related:** nais/grafana-apm-app#133 (SDK PRD, implements this), nais/grafana-apm-app#134
  (platform PRD, implements this), nais/grafana-apm-app#59 (original SDK PRD),
  nais/grafana-apm-app#85 (pipeline gaps), nais/grafana-apm-app#86 (multi-tenancy),
  naiserator `pkg/resourcecreator/frontend/frontend.go`

## Context

`@nais/apm` promises zero-config `init()` on nais. Pilot feedback says
auto-config "doesn't work" or is "hard to work with", and the diagnosis is a
contract with only one side implemented:

- The SDK's primary resolution path is **meta tags the platform never
  writes**. The `nais-app`/`nais-team`/`nais-cluster`/`nais-version`/
  `nais-telemetry-url` tags were *proposed* in the #59 PRD as a contract "to
  be implemented platform-side" — no injection mechanism was ever specified,
  and none exists. Naiserator has no HTML-rewriting machinery at all.
- The platform's actual channel is **consumed by nothing**. Naiserator's
  `spec.frontend.generatedConfig` mounts a `nais.js` ConfigMap
  (`{telemetryCollectorURL, app: {name, version}}`) into the pod and sets
  `NAIS_FRONTEND_TELEMETRY_COLLECTOR_URL` — the SDK reads neither.
- The build-time env path is **structurally broken** for the standard flow:
  `NAIS_*` are pod-runtime vars, absent in CI where bundles are built, and
  the cluster is unknowable at build time anyway — one image deploys to many
  clusters. The collector fallback derives from the cluster name, so it
  collapses with it: bare `init()` on a static SPA resolves no collector,
  silently drops into dev mode, and sends nothing — in production.

Two structural facts constrain any fix:

1. **Field knowability differs.** `app` and `namespace` are code-time
   constants, identical in every environment. `version` is knowable at build
   time (`GITHUB_SHA`). `environment` and `telemetryUrl` are knowable only at
   deploy/runtime — no build-time mechanism can ever supply them.
2. **Nais tenancy is physical isolation.** Each agency runs its own clusters,
   naiserator, and TLD. There is no central injector, registry, or shared
   service across tenants. The per-cluster `telemetryCollectorURL` in each
   tenant's naiserator config is the only value that is tenant-correct *by
   construction*.

Prior-art research (2026-07-12, two surveys: platform runtime-config delivery
across Firebase/Vercel/Netlify/Heroku/CF/K8s/Backstage, and config DX across
Sentry/Datadog/New Relic/Elastic/Faro/OTel — full reports attached to
nais/grafana-apm-app#133) established: platforms that *own the contract* (reserved
URL, versioned payload, blessed consumer SDK — Firebase) scale; folk patterns
where every team invents its own `window.__WHATEVER__` do not. Every
artifact-mutating mechanism surveyed (Heroku sed-into-bundle, Netlify
snippets, Backstage placeholders, CRA buildpack) accumulated documented
breakage. No observability vendor throws on misconfig; the differentiator is
message specificity (Datadog best, Faro — which we wrap — weakest: it
continues init with no transport). New Relic's zero-config model works because
browser config derives from server-side config the platform already owns —
which nais has, unflowed. Firebase's auto-initializing `init.js` died with
the ESM shift while its data-only `init.json` aged fine.

## Decision

1. **Classify fields by knowability, and stop apologizing for constants.**
   The canonical call is `init({ app: 'my-app', namespace: 'my-team' })` —
   two code-time constants in code is correct, not a compromise. `version`
   resolves at build time (`GITHUB_SHA`). Only `environment` and
   `telemetryUrl` need runtime delivery, and all machinery is aimed at them.
2. **The platform owns one versioned, data-only payload** —
   `{schemaVersion, telemetryCollectorURL, app: {name, namespace, version},
   environment}` — generated by each tenant's naiserator. (The SDK exposes
   the collector URL as its `telemetryUrl` option — same value, SDK-side
   name; `fromNaisConfig()` maps between the two.) The payload is
   browser-safe by construction (no secrets; the collector URL is
   public-by-design, abuse control stays server-side at Alloy: CORS
   allowlist, rate limits — the posture of every surveyed vendor). The SDK
   is the blessed consumer; app code never parses the shape itself.
3. **Same payload, multiple transports, chosen per serving architecture:**
   pod env vars (SSR), the ConfigMap-mounted `nais.js`/`nais.json`
   (pod-served SPAs), an ingress-served well-known URL on every app origin
   (the durable end state — covers apps whose HTML is not served or templated by any pod they control), and
   meta tags / a page global as the server→browser bridge **authored by SDK
   helpers, never by the platform**.
4. **Never mutate application artifacts.** No HTML rewriting at ingress or
   sidecar, no bundle rewriting. (Compression/streaming/CSP fragility; the
   breakage record above; no pod exists on CDN; no central injector exists
   across isolated tenants.)
5. **Deliver data, never auto-init.** Config files and endpoints carry
   values; initialization stays in the SDK. (Firebase's v9 lesson.)
6. **Fail soft, loudly, mode-aware.** `init()` never throws. Dev mode
   (console echo, nothing sent) only on genuinely local hosts (`localhost`,
   `127.0.0.1`, `*.local`). On any non-local host that fails resolution:
   `console.error` naming the missing field, what data will not flow, and
   the fix. Resolution must be introspectable (`debug: true` prints the
   per-field source table).
7. **Hostname-derived environment/collector is an interim, nav-tenant-only
   fallback.** No surveyed vendor derives config from `window.location` —
   but vendors own no domains, while a platform owns its ingress topology,
   so it is legitimate as a stopgap that un-breaks static SPAs today. It
   lives behind a tenant-profile interface and is **demoted when the
   well-known config URL (nais/grafana-apm-app#134 phase 3) deploys**. It must
   never be load-bearing for a second tenant.

## Options evaluated

### Platform-side meta-tag/HTML injection — rejected

The #59 PRD's original proposal. Requires ingress or sidecar response
rewriting: detect HTML, decompress gzip/brotli, rewrite `<head>`, recompress,
without breaking streaming, caching, or CSP — on every response for every
app; and it cannot reach CDN-served apps at all (no pod serves the HTML).
Every surveyed platform that mutates artifacts accrued breakage (Netlify
snippets silently skip SSR routes; Backstage's injection-point migration
broke external deployments; Heroku's bundle-sed broke on minifier changes).
Naiserator has no machinery in this direction, and building it would be
platform-wide risk for one SDK's convenience.

### Build-time cluster/env inlining — rejected

`NAIS_CLUSTER_NAME` at build time is structurally impossible: the value does
not exist in CI, and one image serves many clusters. Any docs suggesting it
mislead. Build-time inlining remains correct for exactly one field:
`version` via `GITHUB_SHA`.

### Central tenant registry (domain→collector map shipped in the SDK) — rejected

Contradicts the physical-isolation tenancy model: no central anything exists,
and the SDK cannot enumerate sovereign installations. The per-tenant
naiserator config already carries the correct value; the contract flows it.

### Auto-initializing config script — rejected

A `nais.js` that calls `init()` itself couples the platform artifact to one
SDK generation; Firebase's equivalent could not survive the namespaced→ESM
SDK transition. Data only.

## Consequences

- The SDK gains consumers for every platform channel (generatedConfig shape,
  env var, fetched config URL with pre-init buffering) and per-architecture
  helpers (`<NaisMetaTags />` etc.) — nais/grafana-apm-app#133.
- Naiserator's payload must be completed (`app.namespace`, `environment`,
  `schemaVersion` — all trivially available) and later served at a reserved
  URL — nais/grafana-apm-app#134. Nothing there blocks the nav rollout; the SDK
  work stands alone against what naiserator ships today.
- The README's "the platform injects this tag" claim and the build-time
  cluster suggestion are retracted; "zero-config" is re-scoped to
  "zero-*environment*-config" with two constants in code.
- Multi-tenancy (nais/grafana-apm-app#86) becomes a config exercise: a second
  tenant's naiserator already emits the right payload; only the interim
  hostname fallback is nav-scoped, and it is scheduled for demotion.
- Misconfiguration becomes diagnosable at the console instead of silent,
  which is the direct answer to the pilot feedback.
